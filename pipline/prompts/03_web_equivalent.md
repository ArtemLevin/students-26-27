Промпт «Наглядная математика с генерацией»



Создай один полностью автономный файл `index.html` без внешних зависимостей:



- весь CSS и весь JavaScript должны быть встроены внутрь файла;

- не использовать внешние библиотеки и фреймворки;

- страница должна корректно открываться локально в браузере;

- страница должна быть адаптивной и пригодной для печати.



Назначение:

- это наглядная интерактивная математическая веб-страница;

- материал должен быть понятным, визуально сильным и удобным для самостоятельного изучения;

- решение должно быть не просто правильным, а максимально прозрачным и пошаговым.



Стиль и визуальная подача:

- тёмный академический дизайн;

- аккуратная, современная, выразительная вёрстка;

- хороший визуальный контраст;

- крупные и понятные подписи;

- сильный акцент на наглядности, а не на декоративности.



Структура страницы:

- слева — подробное пошаговое решение;

- справа — крупный интерактивный sticky-блок, который остаётся видимым при прокрутке;

- пользователь должен легко сверять модель с решением по мере чтения;

- все промежуточные выводы должны быть явно подписаны и визуально отделены.



Стандарт обязательной структуры решения:

- каждое решение обязательно строить по схеме:

1) что дано;

2) что означает каждое условие;

3) геометрическая или алгебраическая интерпретация;

4) ключевая идея;

5) пошаговое доведение до ответа;

6) финальная проверка ответа по условию;

7) краткий вывод;

- структура должна быть визуально заметной и легко считываемой;

- переходы между этапами должны быть логичными и явно обозначенными.



Блок «Идея решения»:

- перед подробным решением обязательно добавлять короткий блок «Идея решения»;

- в этом блоке в 2–5 пунктах объяснять общий замысел решения без перегрузки вычислительными деталями;

- этот блок должен помогать понять стратегию до чтения полного разбора.



Математическое оформление:

- не использовать raw LaTeX в отображаемом HTML;

- формулы в готовой странице записывать через Unicode / HTML / обычную математическую вёрстку;

- все обозначения должны быть читаемыми и устойчиво отображаться в браузере.



Визуализация:

- обязательно добавить содержательную математическую визуализацию;

- рисунок/схема/график должен быть связан с решением;

- визуализация должна помогать понять идею, а не просто иллюстрировать ответ;

- по возможности добавлять динамическое обновление рисунка при изменении параметров.



Связь решения и рисунка:

- обеспечивать визуальную связку между шагами решения и элементами рисунка;

- при наведении на ключевой шаг решения по возможности подсвечивать соответствующий объект на рисунке;

- при наведении на объект рисунка по возможности подсвечивать соответствующее место в тексте;

- эта связка должна реально помогать пониманию, а не быть декоративной.



Обозначения и легенда:

- все ключевые математические объекты должны иметь устойчивые обозначения по всей странице;

- одинаковые объекты должны быть оформлены одинаково во всех блоках;

- обязательна легенда обозначений, если на схеме больше одного существенного объекта;

- цвета и подписи должны быть согласованы с текстом решения.



Интерактивность:

- использовать более выраженную интерактивность;

- добавить пошаговые подсказки;

- добавить переключатели элементов схемы;

- сделать так, чтобы пользователь мог включать и выключать важные детали модели;

- при изменении параметров должны обновляться и модель, и ключевые пояснения.



ИНТЕРАКТИВНЫЕ ЯРКИЕ ДИНАМИЧЕСКИЕ ПОДСКАЗКИ:

- Обязательно добавить систему интерактивных подсказок для ключевых элементов управления и сложных концепций;

- Визуальное оформление подсказок:

• анимированные иконки-триггеры (`?`, `!`, `i`) с пульсирующим свечением (`animation: hintPulse`);

• всплывающие окна-поповеры с градиентной обводкой и стрелочкой-указателем;

• позиционирование: `top`, `bottom`, `left`, `right` относительно триггера;

• тёмный фон с яркими акцентами (градиенты cyan/teal);

- Содержание подсказок:

• объяснять назначение элемента управления (селекторы, кнопки генерации, переключатели);

• содержать математические формулы в MathML для наглядности;

• включать интерактивные кнопки действий («Попробовать другой тип», «Сгенерировать сейчас», «Показать пример»);

• давать практические советы и рекомендации;

- Визуальные эффекты:

• ✨ пульсация иконки: `@keyframes hintPulse` с эффектом расширения тени;

• 🎯 подсветка активного элемента: светящаяся рамка `@keyframes borderGlow`;

• 💫 анимация появления: `transform + opacity` transition;

• 🔄 отскок иконки при показе: `@keyframes hintBounce`;

- Поведение системы:

• показ по `mouseenter`/`focus` (на десктопе);

• авто-скрытие через 15 секунд бездействия;

• закрытие по `Escape`, клику вне, или кнопке «×»;

• первый показ важной подсказки через 3 сек после загрузки (только один раз за сессию, использовать `sessionStorage`);

• при переключении режима/типа задачи показывать краткую подсказку об изменениях;

- Доступность:

• `role="tooltip"`, `aria-hidden`, `aria-label` для скринридеров;

• поддержка клавиатуры (focus, Escape, Tab);

• контрастные цвета для читаемости;

• адаптивность под мобильные устройства (упрощённые подсказки на маленьких экранах);

- Минимум 2-3 подсказки на странице:

• возле селектора типа задачи (объяснение различий между типами);

• на кнопке генерации (что происходит при нажатии);

• на сложных элементах управления (зум, переключатели видимости).



Переключение подробности:

- добавить переключатель уровня объяснения:

• кратко,

• стандартно,

• очень подробно;

- при переключении должна меняться степень детализации текста;

- математическая корректность и логика решения при этом должны сохраняться.



Режим пошагового раскрытия:

- добавить режим пошагового открытия решения;

- пользователь должен иметь возможность раскрывать решение по этапам, а не видеть сразу весь разбор;

- в этом режиме структура решения должна оставаться цельной и не распадаться на несвязанные фрагменты.



Зум:

- в формате «Наглядная математика с генерацией» всегда включать зум чертежа;

- обязательно добавить:

• ползунок масштаба,

• кнопку увеличения,

• кнопку уменьшения,

• кнопку сброса масштаба;

- по возможности поддержать масштабирование колесом мыши прямо на чертеже.



Генерация похожих задач:

- обязательно добавить генератор похожих задач;

- каждая сгенерированная задача должна иметь ответ;

- сгенерированную задачу нужно применять к той же интерактивной модели;

- генерация должна быть безопасной и корректной с математической точки зрения;

- генератор должен сохранять тот же математический метод решения;

- генератор не должен порождать некорректные, случайные или вырожденные случаи без специальной пометки;

- по возможности вместе с ответом показывать краткую идею решения;

- генерация должна быть управляемой и понятной пользователю.



Проверка ответа:

- после нахождения ответа обязательно выполнять явную проверку;

- нужно отдельно показать, почему найденные значения подходят;

- нужно отдельно показать, почему других значений нет;

- решение не должно обрываться сразу после получения численного или символьного результата.



Типичные ошибки:

- добавлять отдельный блок «Типичные ошибки»;

- в нём перечислять 3–5 наиболее вероятных неверных ходов именно для данной задачи;

- ошибки должны быть содержательными и связанными с темой, а не общими фразами.



Дополнительно:

- если уместно, добавить мини-квиз или блок самопроверки;

- интерфейс должен быть понятным без дополнительных объяснений;

- код должен быть чистым, автономным и логично организованным.



Техническая устойчивость интерфейса:

- избегать визуальных скачков интерфейса;

- sticky-блок не должен перекрывать текст;

- элементы управления не должны наезжать друг на друга на узких экранах;

- все подписи, формулы и элементы схемы должны корректно отображаться при масштабировании страницы браузером;

- мобильная версия должна оставаться читаемой и управляемой;

- интерфейс не должен ломаться при длинных формулах, многострочных пояснениях и переключении режимов.



Специальное требование для систем уравнений и неравенств:

- в HTML не использовать чрезмерно крупную фигурную скобку для записи системы;

- знак системы должен быть визуально компактным;

- он должен быть аккуратно выровнен по высоте строк;

- он должен стабильно отображаться в браузерах;

- скобка не должна перетягивать на себя внимание и не должна выглядеть выше, чем нужно для двух строк.



Специальное требование для задач по геометрии/аналитической алгебре/векторам:

- фигура должна быть представлена в наглядной изометрии;

- должна быть возможность крутить фигуру;

- линии, которые лежат за видимыми плоскостями, должны динамически пунктириться.



Иконка вкладки (favicon):



- обязательно сгенерируй favicon для страницы;

- favicon должен быть встроен прямо в `index.html`, без отдельных файлов;

- использовать формат SVG через `data:image/svg+xml`;

- favicon должен соответствовать теме страницы и её визуальному стилю;

- иконка должна быть простой, контрастной и хорошо читаемой в маленьком размере;

- избегать мелких деталей, длинного текста и сложных форм;

- желательно использовать 1–2 главных символа темы:

• для алгебры — ось, дробь, знак неравенства, график, x;

• для геометрии — треугольник, окружность, угол, координаты;

- иконка должна хорошо смотреться и в тёмной, и в светлой теме;

- если на странице есть переключение темы, favicon тоже должен визуально оставаться читаемым;

- сгенерируй и сразу вставь строку вида:

`<link rel="icon" href="data:image/svg+xml,...">`

внутрь блока `<head>`;

- не использовать внешние ссылки, отдельные `favicon.ico`, PNG или сторонние ресурсы.



Общее требование:

1)  управление разбором должно скроллиться вместе с чертежом.

2) информация напрямую не относящаяся к учебному материалу на страницу выводиться не должна.

3) если разборов/тем несколько, то создайте панель для переключения содержимого страницы между соответствующими разборами/темами

4) Вся математика должна быть обернута в MathML (это очень строго)

5) должен быть способ переключаться между светлой и темной темой.

6) в интерактивном блоке должна быть возможность двигаться мышкой по схеме
