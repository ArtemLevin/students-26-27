# 4. Основной промпт

Создай один полностью автономный файл `index.html` без внешних зависимостей.

## 4.1. Технические требования к файлу

* весь CSS должен быть встроен внутрь файла;
* весь JavaScript должен быть встроен внутрь файла;
* не использовать внешние библиотеки и фреймворки;
* страница должна корректно открываться локально в браузере;
* страница должна быть адаптивной;
* страница должна быть пригодной для печати.

---

## 4.2. Назначение страницы

Страница должна быть наглядной интерактивной математической веб-страницей.

Материал должен быть:

* понятным;
* визуально сильным;
* удобным для самостоятельного изучения;
* математически корректным;
* пошаговым;
* максимально прозрачным.

Решение должно быть не просто правильным, а методически выстроенным так, чтобы ученик понимал каждое действие.

---

## 4.3. Стиль и визуальная подача

Используй:

* тёмный академический дизайн;
* аккуратную современную вёрстку;
* выразительную визуальную структуру;
* хороший визуальный контраст;
* крупные и понятные подписи;
* сильный акцент на наглядности.

Важно: декоративность не должна мешать учебной ясности.

---

# 5. Структура страницы

Страница должна иметь двухколоночную структуру:

## Левая часть

Слева располагается подробное пошаговое решение.

## Правая часть

Справа располагается крупный интерактивный `sticky`-блок, который остаётся видимым при прокрутке.

Пользователь должен легко сверять интерактивную модель с решением по мере чтения.

Все промежуточные выводы должны быть:

* явно подписаны;
* визуально отделены;
* логически связаны с рисунком или моделью.

---

# 6. Стандарт обязательной структуры решения

Каждое решение обязательно строить по схеме:

1. **Что дано.**
2. **Что означает каждое условие.**
3. **Геометрическая или алгебраическая интерпретация.**
4. **Ключевая идея.**
5. **Пошаговое доведение до ответа.**
6. **Финальная проверка ответа по условию.**
7. **Краткий вывод.**

Требования:

* структура должна быть визуально заметной;
* этапы должны легко считываться;
* переходы между этапами должны быть логичными;
* каждый переход должен быть явно обозначен.

---

# 7. Блок «Идея решения»

Перед подробным решением обязательно добавь короткий блок:

## Идея решения

В этом блоке нужно в 2–5 пунктах объяснить общий замысел решения без перегрузки вычислениями.

Назначение блока:

* показать стратегию;
* подготовить ученика к полному разбору;
* объяснить, почему выбран именно такой метод.

---

# 8. Математическое оформление

## 8.1. Запрет на raw LaTeX

Не использовать raw LaTeX в отображаемом HTML.

Нельзя выводить в интерфейсе выражения вида:

* `\frac{}{}`
* `\sqrt{}`
* `x^{2}`
* `\leq`
* `\begin{cases}`

## 8.2. MathML

Вся математика должна быть обёрнута в **MathML**.

Это строгое требование.

Формулы должны:

* устойчиво отображаться в браузере;
* быть читаемыми;
* не ломать интерфейс;
* корректно масштабироваться.

---

# 9. Визуализация

На странице обязательно должна быть содержательная математическая визуализация.

Это может быть:

* рисунок;
* схема;
* график;
* координатная модель;
* комбинированная модель.

Визуализация должна:

* быть связана с решением;
* помогать понять идею;
* не быть декоративной;
* динамически обновляться при изменении параметров, если это возможно.

---

# 10. Связь решения и рисунка

Обеспечь визуальную связку между шагами решения и элементами рисунка.

Желательно реализовать:

* при наведении на ключевой шаг решения подсвечивается соответствующий объект на рисунке;
* при наведении на объект рисунка подсвечивается соответствующее место в тексте;
* элементы рисунка и элементы решения имеют согласованные обозначения;
* подсветка помогает пониманию, а не является декоративной.

---

# 11. Обозначения и легенда

Все ключевые математические объекты должны иметь устойчивые обозначения по всей странице.

Требования:

* одинаковые объекты оформляются одинаково во всех блоках;
* если на схеме больше одного существенного объекта, обязательна легенда;
* цвета и подписи должны быть согласованы с текстом решения;
* обозначения не должны меняться от блока к блоку.

---

# 12. Интерактивность

Используй выраженную интерактивность.

Обязательно добавь:

* пошаговые подсказки;
* переключатели элементов схемы;
* возможность включать и выключать важные детали модели;
* обновление модели при изменении параметров;
* обновление ключевых пояснений при изменении параметров.

---

# 13. Модальное окно для визуализации

Блок визуализации должен работать как триггер для модального окна.

## Поведение

По клику на основной `canvas` должно открываться `div-overlay` с затемнением фона.

Внутри `overlay` должна рендериться увеличенная копия графика.

## Синхронизация состояния

Изменение параметров в основном окне должно применяться и к модальному окну.

Изменение параметров в модальном окне должно применяться к основному окну.

Например, изменение ползунка масштаба или параметра модели должно синхронно обновлять обе версии.

## Закрытие модального окна

Модальное окно должно закрываться:

* по клику на крестик;
* по клавише `Esc`;
* по клику вне области графика.

# 14. Интерактивные яркие динамические подсказки

Обязательно добавь систему интерактивных подсказок для ключевых элементов управления и сложных концепций.

## 14.1. Визуальное оформление подсказок

Используй:

* анимированные иконки-триггеры: `?`, `!`, `i`;
* пульсирующее свечение через `animation: hintPulse`;
* всплывающие окна-поповеры;
* градиентную обводку;
* стрелочку-указатель;
* позиционирование относительно триггера: `top`, `bottom`, `left`, `right`;
* тёмный фон с яркими акцентами;
* градиенты в стиле `cyan / teal`.

## 14.2. Содержание подсказок

Подсказки должны:

* объяснять назначение элементов управления;
* пояснять селекторы, кнопки генерации, переключатели;
* содержать математические формулы в MathML;
* включать интерактивные кнопки действий:

    * «Попробовать другой тип»;
    * «Сгенерировать сейчас»;
    * «Показать пример»;
* давать практические советы и рекомендации.

## 14.3. Визуальные эффекты

Добавь:

* пульсацию иконки через `@keyframes hintPulse`;
* подсветку активного элемента через светящуюся рамку `@keyframes borderGlow`;
* анимацию появления через `transform + opacity transition`;
* отскок иконки при показе через `@keyframes hintBounce`.

## 14.4. Поведение системы подсказок

Реализуй:

* показ по `mouseenter` и `focus` на десктопе;
* авто-скрытие через 15 секунд бездействия;
* закрытие по `Escape`;
* закрытие по клику вне подсказки;
* закрытие по кнопке `×`;
* первый показ важной подсказки через 3 секунды после загрузки;
* первый показ только один раз за сессию через `sessionStorage`;
* краткую подсказку при переключении режима или типа задачи.

## 14.5. Доступность

Подсказки должны поддерживать:

* `role="tooltip"`;
* `aria-hidden`;
* `aria-label`;
* управление с клавиатуры;
* `focus`;
* `Escape`;
* `Tab`;
* контрастные цвета;
* адаптивность под мобильные устройства.

На маленьких экранах подсказки должны быть упрощёнными.

## 14.6. Минимальное количество подсказок

На странице должно быть минимум 2–3 подсказки:

* возле селектора типа задачи;
* на кнопке генерации;
* на сложных элементах управления:

    * зум;
    * переключатели видимости;
    * параметры модели.

# 15. Переключение подробности

* кратко;
* стандартно;
* очень подробно.

При переключении должна меняться степень детализации текста.

Математическая корректность, логика решения и структура доказательства должны сохраняться при любом режиме.

# 16. Режим пошагового раскрытия

Добавь режим пошагового открытия решения.

Пользователь должен иметь возможность раскрывать решение по этапам, а не видеть сразу весь разбор.

Требования:

* структура решения должна оставаться цельной;
* этапы не должны превращаться в несвязанные фрагменты;
* каждый раскрытый шаг должен быть логически связан с предыдущим.

---

# 17. Зум чертежа

В формате «Наглядная математика с генерацией» всегда включай зум чертежа.

Обязательно добавь:

* ползунок масштаба;
* кнопку увеличения;
* кнопку уменьшения;
* кнопку сброса масштаба.

По возможности поддержи масштабирование колесом мыши прямо на чертеже.

---

# 18. Генерация похожих задач

Обязательно добавь генератор похожих задач.

Каждая сгенерированная задача должна иметь:

* условие;
* ответ;
* краткую идею решения, если это уместно.

Сгенерированную задачу нужно применять к той же интерактивной модели.

Генерация должна быть:

* безопасной;
* математически корректной;
* управляемой;
* понятной пользователю.

Генератор должен:

* сохранять тот же математический метод решения;
* не порождать некорректные случаи;
* не порождать случайные вырожденные случаи без специальной пометки;
* явно показывать, какие параметры были изменены.

---

# 19. Проверка ответа

После нахождения ответа обязательно выполнить явную проверку.

Нужно отдельно показать:

1. почему найденные значения подходят;
2. почему других значений нет.

Решение не должно обрываться сразу после получения численного или символьного результата.

---

# 20. Типичные ошибки

Добавь отдельный блок:

## Типичные ошибки

Ошибки должны быть:

* содержательными;
* связанными с темой;
* конкретными;
* полезными для самопроверки.

Не использовать общие фразы без математического содержания.

# 21. Мини-квиз и самопроверка

Если уместно, добавь:

* мини-квиз;

* вопросы на понимание;
* задания на проверку ключевой идеи.

Интерфейс должен быть понятным без дополнительных объяснений.

# 22. Техническая устойчивость интерфейса

Интерфейс должен быть устойчивым.

Требования:

* избегать визуальных скачков;
* `sticky`-блок не должен перекрывать текст;
* элементы управления не должны наезжать друг на друга на узких экранах;
* все подписи должны корректно отображаться при масштабировании страницы браузером;
* формулы не должны ломать сетку;
* элементы схемы должны корректно масштабироваться;
* мобильная версия должна оставаться читаемой и управляемой;
* интерфейс не должен ломаться при длинных формулах;
* интерфейс не должен ломаться при многострочных пояснениях;
* интерфейс не должен ломаться при переключении режимов.

Код должен быть:

* чистым;
* автономным;
* логично организованным;
* без внешних зависимостей.

---

# 23. Специальное требование для систем уравнений и неравенств

В HTML не использовать чрезмерно крупную фигурную скобку для записи системы.

Требования к знаку системы:

* знак должен быть визуально компактным;
* он должен быть аккуратно выровнен по высоте строк;
* он должен стабильно отображаться в браузерах;
* скобка не должна перетягивать внимание;
* скобка не должна выглядеть выше, чем нужно для двух строк.

---

# 24. Специальное требование для задач по геометрии, аналитической алгебре и векторам

Если задача относится к геометрии, аналитической алгебре или векторам:

* фигура должна быть представлена в наглядной изометрии;
* должна быть возможность крутить фигуру;
* линии, которые лежат за видимыми плоскостями, должны динамически пунктириться.

---

# 25. Favicon

Обязательно сгенерируй favicon для страницы.

Требования:

* favicon должен быть встроен прямо в `index.html`;
* не использовать отдельные файлы;
* использовать формат SVG через `data:image/svg+xml`;
* favicon должен соответствовать теме страницы;
* favicon должен соответствовать визуальному стилю страницы;
* иконка должна быть простой;
* иконка должна быть контрастной;
* иконка должна хорошо читаться в маленьком размере;
* избегать мелких деталей;
* избегать длинного текста;
* избегать сложных форм.

Желательно использовать 1–2 главных символа темы:

### Для алгебры

* ось;
* дробь;
* знак неравенства;
* график;
* `x`.

### Для геометрии

* треугольник;
* окружность;
* угол;
* координаты.

Иконка должна хорошо смотреться и в тёмной, и в светлой теме.

Если на странице есть переключение темы, favicon тоже должен оставаться читаемым.

Сгенерируй и сразу вставь строку вида:

```html
<link rel="icon" href="data:image/svg+xml,...">
```

внутрь блока `<head>`.

Не использовать:

* внешние ссылки;
* отдельные `favicon.ico`;
* PNG;
* сторонние ресурсы.

---

# 26. Общее требование

1. Управление разбором должно скроллиться вместе с чертежом.
2. Информация, напрямую не относящаяся к учебному материалу, на страницу выводиться не должна.
3. Если разборов или тем несколько, создай панель для переключения содержимого страницы между соответствующими разборами или темами.
4. Вся математика должна быть обёрнута в MathML. Это строгое требование.
5. Должен быть способ переключаться между светлой и тёмной темой.
6. В интерактивном блоке должна быть возможность двигаться мышкой по схеме.

---

# 27. Итоговая задача для ИИ-агента

На основе темы и выбранных настроек создай полностью автономный файл `index.html`, который представляет собой интерактивную учебную страницу по математике.

Страница должна объединять:

* теорию;
* пошаговое решение;
* наглядную математическую модель;
* интерактивные подсказки;
